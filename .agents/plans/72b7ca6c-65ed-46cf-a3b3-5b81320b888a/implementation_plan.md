## Proposed Changes

### Data Schema Findings: `formation` vs `coaches`

The previous raw JSON extraction (`eww_staff.json`) returned:
```json
"formation": {
  "serviceLoading": {}
}
```
**Why are the `coaches` missing from the shell?**
In the official schema, the overarching `formation` tag exists when a train matches a known diagram, but the inner `coaches` array is completely populated by real-time Loading Data (often generated by weight sensors/CCTV on the actual carriages).
If the Train Operating Company (TOC) — in this case, SWR — doesn't blast real-time load telemetry for that specific journey (common off-peak or on certain legacy units), Darwin sends `formation` with a blank `serviceLoading` shell and zeroes out the actual `coaches` array. **RDM is simply withholding the arrays because the TOC isn't broadcasting them right now.**

To ensure that the ESP32 handles this structural unpredictability seamlessly, the existing implementation inside `nrRDMDataProvider` gracefully parses whatever is returned. No deep firmware fixes are needed for the parser, but we can verify it cleanly handles the missing key.
